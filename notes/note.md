# 笔记

## 论文研究的问题

FLP 研究分布式系统中的二元共识问题。多个进程各自拥有一个初始值，并通过消息通信，希望最终决定同一个值。一个合格的协议至少要保证两件事：不同进程不能决定不同的值；协议也不能完全忽略输入，永远固定输出某个值。在此基础上，人们还希望即使有一个进程发生故障，系统仍能保证作出决定。

论文用分布式数据库的事务提交说明这个问题的重要性。参与同一事务的多个数据管理进程必须一致决定提交还是回滚。如果一部分提交而另一部分回滚，数据库就可能进入不一致状态。类似的一致决定也存在于复制服务、主节点选举和状态机复制中，因此共识是分布式计算的基础问题。

困难在于：当一个进程很久没有回复时，其他进程无法判断它已经崩溃，还是只是运行得很慢，或者消息仍在网络中延迟。继续等待可能导致协议永不终止；过早决定又可能使不同进程作出冲突决定。这种“崩溃与无限慢无法区分”的现象是理解 FLP 的核心直觉。

## 论文的准确结论

FLP 证明，在完全异步的消息传递系统中，只要允许一个进程停止运行，就不存在能够对所有合法执行都保证终止的确定性共识协议。

更准确地说，对于任何满足一致性和非平凡性要求的确定性协议，都可以找到一种仍然符合系统规则的执行顺序，使得协议一直无法作出决定。这里的重点是“存在一种不终止的执行”，而不是“所有执行都会失败”。现实中的协议可以在很多正常执行中迅速达成共识，但在 FLP 的模型里不能对每一种可能的异步调度都承诺终止。

论文采用的共识要求实际上很弱。它只要求 0 和 1 都有可能在某些运行中成为决定值，并只要求至少一个进程最终决定。现代教材通常还会要求决定值与所有进程的初始输入存在更强联系，并要求所有正确进程最终决定。FLP 连较弱的任务都证明无法保证完成，因此也适用于这些要求更强的协议。

## 系统模型

系统至少包含两个进程。每个进程有一个二元输入寄存器、一个初始为空的输出寄存器、程序状态和无界内部存储。进程一旦决定 0 或 1，就不能再改变输出。进程是确定性的：只要当前状态和收到的消息相同，下一步状态及发出的消息就相同。

进程通过消息通信。已经发送但尚未投递的消息保存在一个全局消息缓冲区中。每次执行接收操作时，系统可以交付一条发给该进程的消息，也可以暂时不交付任何消息。即使缓冲区里已经存在消息，系统仍可以继续延迟它，而且没有固定的延迟上界。不过，如果一个正确进程一直运行并不断尝试接收，那么发给它的每条消息最终都必须到达。

因此，FLP 假设的是“可靠但不及时”的通信：消息不会被损坏或重复，发送给持续运行进程的消息最终会到达，但算法不知道需要等待多久。系统也没有同步时钟，不能仅凭超时可靠地判断某个进程已经崩溃。

一个 configuration，也就是全局配置，包含所有进程的内部状态和消息缓冲区中的全部在途消息。必须记录在途消息，是因为两个进程状态完全相同的系统，如果网络中等待投递的消息不同，未来行为也可能不同。

一次原子步骤只由一个进程执行。该进程接收一条消息或收到“当前没有消息”的结果，随后进行本地计算、更新状态，并发送有限条消息。论文把“哪个进程收到了什么”称为一个 event。由于进程是确定性的，给定当前配置和 event 后，下一个配置是唯一的。

一串可以依次执行的 events 称为 schedule，它产生的实际步骤序列称为 run。有限 schedule 结束后得到的配置称为 reachable configuration；从某个初始配置出发可以到达的配置称为 accessible configuration。这些概念将在后面的二价性证明中使用，因为一个配置的“价”取决于从它出发将来可能到达哪些决定结果。

## 故障与正确性

在一个无限运行中，执行无限多个步骤的进程是 nonfaulty process，只执行有限多个步骤的进程是 faulty process。一个 admissible run 至多包含一个 faulty process，并且所有发给 nonfaulty processes 的消息最终都会被收到。这里是“至多一个故障”，因此也包括没有进程实际故障的情况。

论文把安全性称为 partial correctness。它要求任何可达配置都不能同时出现 0 和 1 两种决定，并且 0、1 都必须分别在某些可达配置中成为可能结果。前一项保证 agreement，后一项排除永远输出固定值的无意义协议。

如果协议满足 partial correctness，并且每个 admissible run 中最终都至少有一个进程决定，论文便称它能够在一个故障下完全正确。主定理说明不存在这样的协议。
