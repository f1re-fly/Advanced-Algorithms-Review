# 问题与系统模型

## 论文研究的问题

FLP 讨论二元共识。多个进程各有一个初始值，通过消息通信决定同一个值。协议不能让不同进程决定不同值，也不能忽略输入并固定输出某个值。即使一个进程停止，协议也要作出决定。

论文以分布式数据库的事务提交为例。参与同一事务的进程必须一致决定提交或回滚，否则数据库会进入不一致状态。复制服务、主节点选举和状态机复制也需要一致决定。

一个进程长时间没有回复时，其他进程无法判断它已经崩溃、运行缓慢，还是消息仍在传输。等待可能导致协议不终止，提前决定可能产生冲突。

## 论文结论

FLP 证明，在完全异步的消息传递系统中，只要允许一个进程停止运行，就不存在能够对所有合法执行都保证终止的确定性共识协议。

任何满足一致性和非平凡性要求的确定性协议，都存在一种符合系统规则但一直无法决定的执行。FLP 证明的是“存在一种不终止的执行”，不是“所有执行都会失败”。

论文只要求 0 和 1 都可能在某些运行中成为决定值，并要求至少一个进程最终决定。现代定义通常还要求决定值与初始输入相关，并要求所有正确进程最终决定。FLP 对较弱的要求已经成立，也适用于要求更强的协议。

## 系统模型

系统至少有两个进程。每个进程包含二元输入、初始为空的输出、程序状态和无界内部存储。进程决定 0 或 1 后不能修改输出。进程是确定性的：当前状态和收到的消息相同时，下一状态和发出的消息也相同。

已发送但未投递的消息保存在全局缓冲区。进程执行接收操作时，系统可以投递一条消息，也可以暂不投递。消息延迟没有上界，但发给持续运行进程的消息最终必须到达。

消息可靠，但投递时间不确定。系统没有同步时钟，不能仅凭超时判断进程已经崩溃。

configuration 包含所有进程的内部状态和缓冲区中的在途消息。在途消息不同，后续执行也可能不同。

一次原子步骤只由一个进程执行。该进程接收一条消息或空结果，然后计算、更新状态并发送有限条消息。论文把“哪个进程收到什么”称为 event。给定配置和 event 后，下一个配置唯一确定。

可依次执行的一串 event 称为 schedule，产生的步骤序列称为 run。有限 schedule 结束后的配置是 reachable configuration；从初始配置可到达的配置是 accessible configuration。一个配置的“价”由它能到达的决定值确定。

## 故障与正确性

在无限运行中，执行无限多个步骤的进程是 nonfaulty process，只执行有限多个步骤的进程是 faulty process。admissible run 至多包含一个 faulty process，且发给 nonfaulty processes 的消息最终都会被接收。“至多一个”也包括没有故障的情况。

partial correctness 要求可达配置中不能同时出现 0 和 1 两种决定，并且 0、1 都能在某些可达配置中成为决定值。前一项保证 agreement，后一项排除固定输出。

如果协议满足 partial correctness，且每个 admissible run 中最终至少有一个进程决定，论文称该协议在一个故障下 totally correct。主定理说明不存在这样的协议。
